# Requerimiento Funcional: RF 15 - Generación de reportes (Plantilla Educativa)

## Explicación del Requerimiento (Genérico)
Este requerimiento define el flujo de comunicación estándar para consultar y listar reportes de manera paginada, ordenada y filtrada. La paginación evita sobrecargar la memoria del servidor y de la red al retornar solo un subconjunto de los datos en cada solicitud, permitiendo al usuario navegar a través de páginas utilizando controles visuales.

---

## Estructura del Flujo de Comunicación (Plantilla educativa)

### 1. Frontend (React)
El flujo se inicia en el componente React `[ComponenteReact]`. Éste mantiene en su estado local la página actual, el tamaño de la página y el filtro de búsqueda seleccionado, y realiza la consulta en un efecto de React.

```javascript
// [ComponenteReact].jsx
import React, { useEffect, useState } from 'react';

export const ComponenteReporte = () => {
  const [reportes, setReportes] = useState([]);
  const [pagina, setPagina] = useState(0);
  const [totalPaginas, setTotalPaginas] = useState(0);
  const [filtro, setFiltro] = useState('');

  useEffect(() => {
    const cargarReportes = async () => {
      try {
        const respuesta = await fetchReportesPaginados(pagina, 10, filtro);
        setReportes(respuesta.content);
        setTotalPaginas(respuesta.totalPages);
      } catch (error) {
        console.error('Error al cargar reportes:', error);
      }
    };
    cargarReportes();
  }, [pagina, filtro]);

  return (
    <div>
      <input type="text" placeholder="Filtrar reporte..." onChange={(e) => { setFiltro(e.target.value); setPagina(0); }} />
      <ul>
        {reportes.map(rep => <li key={rep.id}>{rep.titulo}</li>)}
      </ul>
      <button disabled={pagina === 0} onClick={() => setPagina(pagina - 1)}>Anterior</button>
      <span>Página {pagina + 1} de {totalPaginas}</span>
      <button disabled={pagina >= totalPaginas - 1} onClick={() => setPagina(pagina + 1)}>Siguiente</button>
    </div>
  );
};
```

### 2. Red (HTTP Call)
La petición de red viaja como una llamada GET con parámetros de consulta (`query params`).

- **Método HTTP:** `GET`
- **Ruta/URL:** `[EndpointUrl]?page={pagina}&size={tamanio}&filtro={texto}` (Ejemplo: `/api/v1/reportes?page=0&size=10&filtro=academico`)
- **Cabeceras (Headers):** `Accept: application/json`
- **Cuerpo (Request Body):** Vacío.

### 3. Controller (Spring Boot)
El controlador `[ControllerClass]` recibe los parámetros de la URL y los empaqueta en una clase de soporte de paginación (`Pageable`) de Spring Framework.

```java
// [ControllerClass].java
@RestController
@RequestMapping("/api/v1/reportes")
public class ControllerClass {

    private final ServiceClass serviceClass;

    public ControllerClass(ServiceClass serviceClass) {
        this.serviceClass = serviceClass;
    }

    @GetMapping
    public ResponseEntity<Page<JPAEntity>> listarReportes(
            @RequestParam(defaultValue = "0") int page,
            @RequestParam(defaultValue = "10") int size,
            @RequestParam(required = false) String filtro) {

        Pageable pageable = PageRequest.of(page, size, Sort.by("id").descending());
        Page<JPAEntity> resultado = serviceClass.obtenerReportes(filtro, pageable);
        return ResponseEntity.ok(resultado);
    }
}
```

### 4. Service
El servicio `[ServiceClass]` ejecuta la consulta correspondiente empleando el repositorio y pasando el objeto de paginación.

```java
// [ServiceClass].java
@Service
public class ServiceClass {

    private final RepositoryInterface repositoryInterface;

    public ServiceClass(RepositoryInterface repositoryInterface) {
        this.repositoryInterface = repositoryInterface;
    }

    @Transactional(readOnly = true)
    public Page<JPAEntity> obtenerReportes(String filtro, Pageable pageable) {
        if (filtro != null && !filtro.trim().isEmpty()) {
            return repositoryInterface.findByTituloContainingIgnoreCase(filtro, pageable);
        }
        return repositoryInterface.findAll(pageable);
    }
}
```

### 5. Repository & Entity
La interfaz `[RepositoryInterface]` extiende de `JpaRepository` y define un método que acepta `Pageable` y retorna una estructura `Page`.

```java
// [RepositoryInterface].java
@Repository
public interface RepositoryInterface extends JpaRepository<JPAEntity, Long> {
    Page<JPAEntity> findByTituloContainingIgnoreCase(String filtro, Pageable pageable);
}

// [JPAEntity].java
@Entity
@Table(name = "tabla_reportes")
public class JPAEntity {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "titulo", nullable = false)
    private String titulo;

    // Getters y Setters
}
```

### 6. Return Path
El flujo de retorno es el siguiente:
1. **Base de Datos:** Ejecuta la consulta SQL con cláusula `LIMIT` / `OFFSET` y una consulta secundaria de conteo (`COUNT`).
2. **Repository:** Recibe el conjunto de registros y genera el objeto de tipo `Page<JPAEntity>`.
3. **Service:** Devuelve el objeto `Page` al controlador.
4. **Controller:** Retorna la respuesta HTTP con estado `200 OK` y el cuerpo en formato JSON conteniendo la paginación (`content`, `totalPages`, `totalElements`, etc.).
5. **Red:** Viaja la estructura JSON por la red hacia el frontend.
6. **Frontend (React):** `fetchReportesPaginados` resuelve la promesa y actualiza el estado local de los reportes y metadatos de paginación en `[ComponenteReact]`.
