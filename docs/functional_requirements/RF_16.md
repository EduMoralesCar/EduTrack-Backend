# Requerimiento Funcional: RF 16 - Notificaciones académicas (Plantilla Educativa)

## Explicación del Requerimiento (Genérico)
Este requerimiento define el flujo de comunicación estándar para realizar una operación transaccional compleja que requiere ACID (Atomicidad, Consistencia, Aislamiento y Durabilidad). Se utiliza cuando una única solicitud del usuario genera modificaciones en múltiples entidades o tablas en PostgreSQL de forma simultánea. Si una parte del proceso falla, todo el flujo debe revertirse (hacer Rollback) para evitar inconsistencias en el estado de la base de datos.

---

## Estructura del Flujo de Comunicación (Plantilla educativa)

### 1. Frontend (React)
El flujo comienza en el componente React `[ComponenteReact]`, el cual envía un conjunto estructurado de datos que incluye información principal y sus ítems de detalle correspondientes.

```javascript
// [ComponenteReact].jsx
import React, { useState } from 'react';

export const ComponenteNotificaciones = () => {
  const [notificacion, setNotificacion] = useState({
    asunto: '',
    mensaje: '',
    destinatariosIds: []
  });

  const handleEnviarNotificacion = async () => {
    try {
      const respuesta = await fetchEnviarNotificacion(notificacion);
      console.log('Notificación enviada y registrada con éxito:', respuesta);
    } catch (error) {
      console.error('Error al enviar la notificación transaccional:', error);
    }
  };

  return (
    <div>
      {/* Elementos del formulario para asunto, mensaje, etc. */}
      <button onClick={handleEnviarNotificacion}>Enviar Notificaciones</button>
    </div>
  );
};
```

### 2. Red (HTTP Call)
Los datos estructurados e interactivos se transmiten al backend mediante una petición HTTP POST.

- **Método HTTP:** `POST`
- **Ruta/URL:** `[EndpointUrl]` (Ejemplo: `/api/v1/notificaciones/enviar`)
- **Cabeceras (Headers):** `Content-Type: application/json`
- **Cuerpo (Request Body):**
```json
{
  "asunto": "Aviso Urgente",
  "mensaje": "Reunión de padres de familia",
  "destinatariosIds": [10, 11, 12]
}
```

### 3. Controller (Spring Boot)
El controlador `[ControllerClass]` expone el endpoint POST, valida los datos del DTO recibido y los envía al servicio transaccional.

```java
// [ControllerClass].java
@RestController
@RequestMapping("/api/v1/notificaciones")
public class ControllerClass {

    private final ServiceClass serviceClass;

    public ControllerClass(ServiceClass serviceClass) {
        this.serviceClass = serviceClass;
    }

    @PostMapping("/enviar")
    public ResponseEntity<ResponseDTO> enviarNotificaciones(@Valid @RequestBody NotificacionDTO dto) {
        ResponseDTO respuesta = serviceClass.procesarEnvioTransaccional(dto);
        return ResponseEntity.ok(respuesta);
    }
}
```

### 4. Service
El servicio `[ServiceClass]` utiliza la anotación `@Transactional` en su método principal para garantizar que todas las inserciones en la base de datos se ejecuten dentro de la misma transacción de PostgreSQL. Si ocurre alguna excepción, se revierte todo de manera automática.

```java
// [ServiceClass].java
@Service
public class ServiceClass {

    private final RepositoryA repositoryA;
    private final RepositoryB repositoryB;

    public ServiceClass(RepositoryA repositoryA, RepositoryB repositoryB) {
        this.repositoryA = repositoryA;
        this.repositoryB = repositoryB;
    }

    @Transactional
    public ResponseDTO procesarEnvioTransaccional(NotificacionDTO dto) {
        // 1. Guardar registro de la notificación principal (Cabecera)
        EntidadCabecera cabecera = new EntidadCabecera();
        cabecera.setAsunto(dto.getAsunto());
        cabecera.setMensaje(dto.getMensaje());
        EntidadCabecera cabeceraGuardada = repositoryA.save(cabecera);

        // 2. Iterar e insertar destinatarios (Detalle)
        for (Long destinatarioId : dto.getDestinatariosIds()) {
            EntidadDetalle detalle = new EntidadDetalle();
            detalle.setNotificacionId(cabeceraGuardada.getId());
            detalle.setDestinatarioId(destinatarioId);
            detalle.setEstado("PENDIENTE");

            // Simulación de validación de negocio que podría fallar
            if (destinatarioId <= 0) {
                // Al lanzar RuntimeException, se realiza ROLLBACK de toda la transacción
                throw new IllegalArgumentException("ID de destinatario inválido: " + destinatarioId);
            }
            repositoryB.save(detalle);
        }

        return new ResponseDTO(cabeceraGuardada.getId(), "PROCESADO");
    }
}
```

### 6. Return Path
El flujo de retorno es el siguiente:
1. **Base de Datos (Éxito):** Ejecuta el `COMMIT` en PostgreSQL y consolida los cambios de todas las tablas de forma permanente.
2. **Base de Datos (Fallo):** Si el servicio arroja una excepción, ejecuta el `ROLLBACK` y revierte todas las inserciones/actualizaciones en la BD.
3. **Service:** Si fue exitosa, retorna el `ResponseDTO`. En caso de fallo, la excepción escala hasta el manejador global.
4. **Controller:** Retorna la respuesta HTTP `200 OK` con la información del proceso exitoso, o bien el manejador global retorna un estado HTTP de error (ej. `400 Bad Request`).
5. **Red:** Envía la respuesta HTTP al cliente frontend.
6. **Frontend (React):** El cliente recibe el resultado exitoso y actualiza el estado de la aplicación, o despliega un mensaje de error si la transacción falló.
