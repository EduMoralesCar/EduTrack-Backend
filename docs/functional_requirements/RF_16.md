# Requerimiento Funcional: [RF_16 - Nombre del Requerimiento de Operación Compleja] (Plantilla Educativa)

## Explicación del Requerimiento (Genérico)
Esta plantilla describe el flujo de comunicación estándar para una **operación transaccional compleja** que involucra múltiples entidades y requiere ACID (Atomicidad, Consistencia, Aislamiento y Durabilidad). Si alguna de las operaciones falla, todo el conjunto de operaciones debe revertirse (Rollback), impidiendo inconsistencias en la base de datos PostgreSQL.

---

## Estructura del Flujo de Comunicación (Plantilla educativa)

### 1. Frontend (React)
El flujo comienza en el componente React `[ComponenteReactProceso]` al iniciar una transacción compleja que requiere enviar múltiples datos relacionados.

```javascript

import React, { useState } from 'react';

export const ComponenteProceso = () => {
  const [pasoActual, setPasoActual] = useState(1);

  return (
    <div>

      {/* Detalle de resumen a procesar */}

};
```

### 2. Red (HTTP Call)
La llamada de red se realiza mediante una petición HTTP POST con una estructura de datos compleja.

- **Método HTTP:** `POST`
- **Ruta/URL:** `[EndpointUrl]/procesar` (Ejemplo: `/api/v1/operaciones/procesar`)
- **Cabeceras (Headers):** `Content-Type: application/json`
- **Cuerpo (Request Body):** JSON anidado que contiene la información de la cabecera y un array de ítems relacionados.

```json
{
  "cabeceraId": 45,
  "detalles": [
    { "recursoId": 1, "cantidad": 5 },
    { "recursoId": 2, "cantidad": 10 }
  ]
}
```

### 3. Controller (Spring Boot)
El controlador `[ControllerClass]` expone el endpoint y mapea el cuerpo complejo a un DTO de solicitud.

```java
// [ControllerClass].java
@RestController
@RequestMapping("/api/v1/operaciones")
public class ControllerClass {

    private final ServiceClass serviceClass;

    public ControllerClass(ServiceClass serviceClass) {
        this.serviceClass = serviceClass;
    }

    @PostMapping("/procesar")
    public ResponseEntity<ResponseDTO> procesarOperacion(@Valid @RequestBody TransaccionDTO dto) {
        ResponseDTO respuesta = serviceClass.ejecutarTransaccionCompleta(dto);
        return ResponseEntity.ok(respuesta);
    }
}
```

### 4. Service
El servicio `[ServiceClass]` tiene la anotación `@Transactional` en su método de lógica de negocio principal para asegurar que todas las llamadas de repositorio se agrupen en la misma transacción y se haga rollback si ocurre alguna excepción en tiempo de ejecución.

```java
// [ServiceClass].java
@Service
public class ServiceClass {

    private final RepositoryA repositoryA;
    private final RepositoryB repositoryB;

    public ServiceClass(RepositoryA repositoryA, RepositoryB repositoryB) {
        this.repositoryA = repositoryA;
        this.repositoryB = repositoryB;
    }

    @Transactional
    public ResponseDTO ejecutarTransaccionCompleta(TransaccionDTO dto) {
        // 1. Guardar o actualizar la entidad principal
        EntidadCabecera cabecera = new EntidadCabecera();
        cabecera.setEstado("PROCESADO");
        EntidadCabecera cabeceraGuardada = repositoryA.save(cabecera);

        // 2. Iterar y actualizar o registrar registros relacionados (Entidad Detalle)
        for (DetalleDTO det : dto.getDetalles()) {
            EntidadDetalle detalle = new EntidadDetalle();
            detalle.setCabeceraId(cabeceraGuardada.getId());
            detalle.setRecursoId(det.getRecursoId());
            detalle.setCantidad(det.getCantidad());
            
            // Simulación de validación de negocio
            if (det.getCantidad() <= 0) {
                // Esta excepción causa un ROLLBACK automático de toda la transacción
                throw new IllegalArgumentException("Cantidad inválida, abortando operación.");
            }
            repositoryB.save(detalle);
        }

        return new ResponseDTO(cabeceraGuardada.getId(), "EXITOSO");
    }
}
```

### 5. Repository & Entity
Las interfaces y clases JPA que mapean el modelo relacional a PostgreSQL.

```java
// [RepositoryA].java & [RepositoryB].java
@Repository
public interface RepositoryA extends JpaRepository<EntidadCabecera, Long> {}

@Repository
public interface RepositoryB extends JpaRepository<EntidadDetalle, Long> {}

// [EntidadCabecera].java
@Entity
@Table(name = "tabla_cabeceras")
public class EntidadCabecera {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String estado;
}

// [EntidadDetalle].java
@Entity
@Table(name = "tabla_detalles")
public class EntidadDetalle {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private Long cabeceraId;
    private Long recursoId;
    private int cantidad;
}
```

### 6. Return Path
El flujo de retorno es el siguiente:
1. **Base de Datos (Transacción Exitosa):** Confirma todas las inserciones y actualizaciones (`COMMIT`).
2. **Base de Datos (Transacción Fallida):** Revierte todos los cambios realizados en el transcurso del método (`ROLLBACK`), dejando la BD en su estado original.
3. **Service:** Si fue exitosa, retorna el `ResponseDTO`. Si falló, la excepción escala.
4. **Controller:** Devuelve una respuesta HTTP `200 OK` con el `ResponseDTO` en caso de éxito, o escala el error para que un controlador global de excepciones devuelva `400 Bad Request` o `500 Internal Server Error`.
5. **Red:** Envía la respuesta serializada al cliente.
6. **Frontend (React):** Actúa en base a la respuesta, ya sea confirmando al usuario la finalización del proceso o mostrando un error explicativo del motivo del fallo de la transacción.
