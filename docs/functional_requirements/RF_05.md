# Requerimiento Funcional: RF 05 - Asignación de docentes (Plantilla Educativa)

Este documento sirve como guía educativa paso a paso para comprender cómo se modela y ejecuta una actualización de relación directa (Foreign Key o Clave Foránea) entre dos entidades en la base de datos PostgreSQL utilizando Spring Boot y React.

---

## Explicación del Requerimiento (Genérico)
El objetivo de este requerimiento es asignar o vincular un registro de tipo "Responsable/Docente" a una entidad secundaria de tipo "Grupo/Sección/Tarea" (Relación Uno a Muchos).
- El sistema debe verificar que la entidad que se asigna cumpla con las precondiciones necesarias (por ejemplo: rol correcto, estado activo, sin conflictos de horarios o sobrecarga).
- La operación actualiza directamente la clave foránea en la tabla del elemento asignado.

---

## Estructura del Flujo de Comunicación (Plantilla educativa)

### 1. Frontend (React)
El componente de React `[ComponenteReact]` permite seleccionar una sección y luego seleccionar un docente idóneo de una lista para actualizar la asignación mediante un método PUT o PATCH.

```javascript
// Ejemplo genérico en React ([ComponenteReact].jsx)
import React, { useState, useEffect } from 'react';

export const [ComponenteReact] = ({ sectionId }) => {
  const [teachers, setTeachers] = useState([]);
  const [selectedTeacherId, setSelectedTeacherId] = useState('');
  const [success, setSuccess] = useState(false);

  useEffect(() => {
    // Carga la lista de docentes disponibles al cargar el componente
    fetchTeachers();
  }, []);

  const fetchTeachers = async () => {
    const response = await fetch('/api/teachers/active', {
      headers: { 'Authorization': 'Bearer ' + localStorage.getItem('token') }
    });
    setTeachers(await response.json());
  };

  const handleAssign = async (e) => {
    e.preventDefault();
    setSuccess(false);

    try {
      const response = await fetch(`[EndpointUrl]/assign-teacher`, {
        method: 'PATCH',
        headers: {
          'Content-Type': 'application/json',
          'Authorization': 'Bearer ' + localStorage.getItem('token')
        },
        body: JSON.stringify({ teacherId: selectedTeacherId })
      });

      if (response.ok) {
        setSuccess(true);
      }
    } catch (error) {
      console.error('Error al realizar asignación', error);
    }
  };

  return (
    <form onSubmit={handleAssign}>
      <h4>Asignar Docente Responsable</h4>
      <select value={selectedTeacherId} onChange={e => setSelectedTeacherId(e.target.value)}>
        <option value="">Seleccione un docente...</option>
        {teachers.map(t => (
          <option key={t.id} value={t.id}>{t.name}</option>
        ))}
      </select>
      <button type="submit">Guardar Asignación</button>
      {success && <p style={{ color: 'green' }}>Docente asignado con éxito.</p>}
    </form>
  );
};
```

### 2. Red (HTTP Call)
Se efectúa una petición de tipo patch/update a un recurso específico pasando el ID del recurso que se va a asociar:

- **Método**: `PATCH` (o `PUT` según convención)
- **URL**: `[EndpointUrl]/assign-teacher` (ej. `http://localhost:8080/api/sections/12/assign-teacher`)
- **Cabeceras**:
  - `Content-Type: application/json`
  - `Authorization: Bearer [TokenJWT]`
- **Cuerpo (Payload JSON)**:
  ```json
  {
    "teacherId": 8
  }
  ```

### 3. Controller (Spring Boot)
El backend expone un endpoint para recibir la solicitud, extrayendo el ID del recurso principal de la ruta (`@PathVariable`) y el ID del recurso a asociar en el cuerpo (`@RequestBody`).

```java
// Ejemplo genérico en Spring Boot ([ControllerClass].java)
package com.example.generic.controller;

import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/api/sections")
public class [ControllerClass] {

    private final [ServiceClass] sectionService;

    public [ControllerClass]([ServiceClass] sectionService) {
        this.sectionService = sectionService;
    }

    @PatchMapping("/{sectionId}/assign-teacher")
    public ResponseEntity<?> assignTeacher(@PathVariable Long sectionId, 
                                           @RequestBody [AssignTeacherDTO] dto) {
        [SectionDTO] updatedSection = sectionService.assignTeacher(sectionId, dto.getTeacherId());
        return ResponseEntity.ok(updatedSection);
    }
}
```

### 4. Service
En `[ServiceClass]` se implementan los checks de negocio para la asignación y se realiza el cambio.

```java
// Ejemplo genérico en Spring Boot ([ServiceClass].java)
package com.example.generic.service;

import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

@Service
public class [ServiceClass] {

    private final [SectionRepositoryInterface] sectionRepository;
    private final [UserRepositoryInterface] userRepository; // Asumiendo los profesores son usuarios

    public [ServiceClass]([SectionRepositoryInterface] sectionRepository, 
                          [UserRepositoryInterface] userRepository) {
        this.sectionRepository = sectionRepository;
        this.userRepository = userRepository;
    }

    @Transactional
    public [SectionDTO] assignTeacher(Long sectionId, Long teacherId) {
        // 1. Validar que la sección exista
        [JPAEntity] section = sectionRepository.findById(sectionId)
                .orElseThrow(() -> new RuntimeException("La sección no existe"));

        // 2. Validar que el profesor a asignar exista
        [UserEntity] teacher = userRepository.findById(teacherId)
                .orElseThrow(() -> new RuntimeException("El docente no existe"));

        // 3. Regla de Negocio: Validar que el rol de la entidad sea el de docente
        if (!"TEACHER".equals(teacher.getRole())) {
            throw new RuntimeException("El usuario seleccionado debe tener el rol de docente");
        }

        // 4. Regla de Negocio: Validar que el docente esté activo
        if (!teacher.isActive()) {
            throw new RuntimeException("El docente seleccionado no está activo");
        }

        // 5. Modificar la relación
        section.setTeacher(teacher);

        // 6. Guardar en base de datos
        [JPAEntity] updatedSection = sectionRepository.save(section);

        return convertToDTO(updatedSection);
    }

    private [SectionDTO] convertToDTO([JPAEntity] entity) {
        [SectionDTO] dto = new [SectionDTO]();
        dto.setId(entity.getId());
        dto.setName(entity.getName());
        if (entity.getTeacher() != null) {
            dto.setTeacherName(entity.getTeacher().getUsername());
        }
        return dto;
    }
}
```

### 6. Return Path
El flujo de retorno viaja en sentido inverso:

1. **PostgreSQL** actualiza el campo de la clave foránea en la tabla correspondiente y confirma la operación.
2. **`[SectionRepositoryInterface]`** retorna la entidad actualizada.
3. **`[ServiceClass]`** mapea la entidad al DTO que incluye los datos actualizados del profesor asignado.
4. **`[ControllerClass]`** envía la respuesta HTTP `200 OK` con el JSON formateado al cliente.
5. **Red**: El cliente recibe el paquete JSON de respuesta.
6. **Frontend (React)**: El frontend captura la resolución asíncrona exitosa de la petición y activa el estado de éxito (`setSuccess(true)`), actualizando dinámicamente la vista para reflejar el cambio del docente asignado al grupo en la interfaz.
