# Requerimiento Funcional: RF 04 - Gestión de secciones y matrículas (Plantilla Educativa)

Este documento sirve como guía educativa paso a paso para comprender cómo se gestionan y persisten las relaciones complejas de negocio (Uno a Muchos y Muchos a Muchos) que vinculan múltiples tablas en una base de datos relacional (PostgreSQL).

---

## Explicación del Requerimiento (Genérico)
El objetivo de este requerimiento es modelar la relación asociativa entre dos o más entidades independientes en el sistema.
- Por ejemplo, un **Curso** se subdivide en múltiples **Secciones** (Relación Uno a Muchos).
- A su vez, los **Estudiantes** se inscriben o matriculan en estas secciones específicas, requiriendo una tabla asociativa intermedia con datos de control adicionales como la fecha de inscripción y el estado (Relación Muchos a Muchos representada por una entidad asociativa intermedia).
- El sistema debe controlar reglas de negocio complejas antes de guardar, como verificar si existe cupo/capacidad en la sección y asegurar que los estados de ambas entidades sean correctos.

## Estructura del Flujo de Comunicación (Plantilla educativa)

### 1. Frontend (React)
El componente `[ComponenteReact]` permite al usuario administrador seleccionar una sección y un estudiante activo desde listas dinámicas y enviar una petición POST para crear la asociación (matrícula).

```javascript
// Ejemplo genérico en React ([ComponenteReact].jsx)
import React, { useState, useEffect } from 'react';

export const [ComponenteReact] = () => {
  const [students, setStudents] = useState([]);
  const [sections, setSections] = useState([]);
  const [selection, setSelection] = useState({ studentId: '', sectionId: '' });
  const [statusMsg, setStatusMsg] = useState('');

  useEffect(() => {
    // Cargar listas iniciales para los selectors desplegables
    loadInitialData();
  }, []);

  const loadInitialData = async () => {
    const headers = { 'Authorization': 'Bearer ' + localStorage.getItem('token') };
    const resStudents = await fetch('/api/students/active', { headers });
    const resSections = await fetch('/api/sections/available', { headers });
    setStudents(await resStudents.json());
    setSections(await resSections.json());
  };

  const handleEnrollSubmit = async (e) => {
    e.preventDefault();
    if (!selection.studentId || !selection.sectionId) return;

    const response = await fetch('[EndpointUrl]', {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': 'Bearer ' + localStorage.getItem('token')
      },
      body: JSON.stringify(selection)
    });

    if (response.ok) {
      setStatusMsg('Asociación (matrícula) registrada exitosamente.');
    } else {
      const err = await response.json();
      setStatusMsg('Error: ' + err.message);
    }
  };

  return (
    <form onSubmit={handleEnrollSubmit}>
      <h3>Inscripción de Estudiantes a Secciones</h3>
      <select onChange={e => setSelection({...selection, studentId: e.target.value})}>
        <option value="">Seleccione Estudiante</option>
        {students.map(s => <option key={s.id} value={s.id}>{s.fullName}</option>)}
      </select>

      <select onChange={e => setSelection({...selection, sectionId: e.target.value})}>
        <option value="">Seleccione Sección</option>
        {sections.map(sec => <option key={sec.id} value={sec.id}>{sec.name}</option>)}
      </select>

      <button type="submit">Vincular Entidades</button>
      {statusMsg && <p>{statusMsg}</p>}
    </form>
  );
};
```

### 2. Red (HTTP Call)
Se envía un payload plano con los identificadores de los recursos que se van a relacionar:

- **Método**: `POST`
- **URL**: `[EndpointUrl]` (ej. `http://localhost:8080/api/enrollments`)
- **Cabeceras (Headers)**:
  - `Content-Type: application/json`
  - `Authorization: Bearer [TokenJWT]`
- **Cuerpo (Payload JSON)**:
  ```json
  {
    "studentId": 45,
    "sectionId": 12
  }
  ```

### 3. Controller (Spring Boot)
El controlador `[ControllerClass]` expone el servicio de asociación, recibiendo los IDs y enviándolos a la capa lógica.

```java
// Ejemplo genérico en Spring Boot ([ControllerClass].java)
package com.example.generic.controller;

import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/api/enrollments")
public class [ControllerClass] {

    private final [ServiceClass] enrollmentService;

    public [ControllerClass]([ServiceClass] enrollmentService) {
        this.enrollmentService = enrollmentService;
    }

    @PostMapping
    public ResponseEntity<?> enrollStudent(@RequestBody [EnrollmentRequestDTO] dto) {
        [EnrollmentResponseDTO] result = enrollmentService.enroll(dto.getStudentId(), dto.getSectionId());
        return new ResponseEntity<>(result, HttpStatus.CREATED);
    }
}
```

### 4. Service
En esta capa, `[ServiceClass]` implementa las validaciones complejas de integridad transaccional: comprueba cupos, busca las dos entidades en la base de datos, inicializa la entidad asociativa y la guarda.

```java
// Ejemplo genérico en Spring Boot ([ServiceClass].java)
package com.example.generic.service;

import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

@Service
public class [ServiceClass] {

    private final [EnrollmentRepositoryInterface] enrollmentRepository;
    private final [StudentRepositoryInterface] studentRepository;
    private final [SectionRepositoryInterface] sectionRepository;

    public [ServiceClass]([EnrollmentRepositoryInterface] enrollmentRepository,
                          [StudentRepositoryInterface] studentRepository,
                          [SectionRepositoryInterface] sectionRepository) {
        this.enrollmentRepository = enrollmentRepository;
        this.studentRepository = studentRepository;
        this.sectionRepository = sectionRepository;
    }

    @Transactional
    public [EnrollmentResponseDTO] enroll(Long studentId, Long sectionId) {
        // 1. Obtener y validar la sección de destino
        [SectionEntity] section = sectionRepository.findById(sectionId)
                .orElseThrow(() -> new RuntimeException("La sección no existe"));

        // 2. Obtener y validar el estudiante
        [StudentEntity] student = studentRepository.findById(studentId)
                .orElseThrow(() -> new RuntimeException("El estudiante no existe"));

        // 3. Regla de Negocio: Verificar cupo disponible
        long currentEnrolled = enrollmentRepository.countBySectionId(sectionId);
        if (currentEnrolled >= section.getCapacity()) {
            throw new RuntimeException("No quedan vacantes disponibles en la sección");
        }

        // 4. Regla de Negocio: Evitar doble inscripción activa
        if (enrollmentRepository.existsByStudentIdAndSectionId(studentId, sectionId)) {
            throw new RuntimeException("El estudiante ya está inscrito en esta sección");
        }

        // 5. Crear la entidad intermedia (Matrícula)
        [JPAEntity] enrollment = new [JPAEntity]();
        enrollment.setStudent(student);
        enrollment.setSection(section);
        enrollment.setStatus("ACTIVE");

        // 6. Guardar en base de datos
        [JPAEntity] saved = enrollmentRepository.save(enrollment);

        return convertToDTO(saved);
    }

    private [EnrollmentResponseDTO] convertToDTO([JPAEntity] entity) {
        return new [EnrollmentResponseDTO](
            entity.getId(),
            entity.getStudent().getFullName(),
            entity.getSection().getName(),
            entity.getStatus()
        );
    }
}
```

### 5. Repository & Entity
La representación relacional en Java usa `@ManyToOne` para apuntar a los extremos de la relación.

```java
// Entidad JPA Intermedia ([JPAEntity].java)
package com.example.generic.model;

import jakarta.persistence.*;

@Entity
@Table(name = "[DatabaseTable_matriculas]")
public class [JPAEntity] {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    // Relación de muchos a uno con Estudiante
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "student_id", nullable = false)
    private [StudentEntity] student;

    // Relación de muchos a uno con Sección
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "section_id", nullable = false)
    private [SectionEntity] section;

    @Column(nullable = false)
    private String status; // Ej. ACTIVE, INACTIVE, COMPLETED

    // Getters y Setters...
}
```

### 6. Return Path
Flujo de respuesta de la operación transaccional:

1. **PostgreSQL** ejecuta la inserción del nuevo registro en la tabla intermedia y actualiza el contador/índice de relaciones.
2. **`[EnrollmentRepositoryInterface]`** retorna la instancia física de la relación creada.
3. **`[ServiceClass]`** confirma la transacción correcta y mapea el resultado a un DTO plano conteniendo los nombres descriptivos de ambas partes de la relación.
4. **`[ControllerClass]`** emite una respuesta HTTP `201 Created` con el JSON de respuesta.
5. **Red**: El paquete de datos de confirmación es enviado.
6. **Frontend (React)**: El manejador de la promesa `fetch` lee el JSON del backend, actualiza el mensaje de estado y limpia los campos del formulario o actualiza el catálogo visible en pantalla con los datos reales devueltos de la relación.
