# RF 04: Gestión de secciones y matrículas

- **Explicación del Requerimiento**:
  Este requerimiento permite la organización académica del alumnado a través de dos entidades principales:
  1. **Secciones**: Subdivisiones de un curso en un determinado periodo (ej. Sección "A" del curso "INF-401" para el periodo "2026-I").
  2. **Matrículas**: Asociación de un estudiante activo (rol `STUDENT`) a una sección específica, controlando adicionalmente su estado académico (`ENROLLED` / ACTIVO, `AT_RISK` / OBSERVADO, `WITHDRAWN` / RETIRADO).
  La creación de secciones y el proceso de matrícula de estudiantes están reservados exclusivamente para el rol `ADMIN`.

- **Flujo de Código Paso a Paso**:

  ## Flujo A: Creación de Secciones

  ### 1. Interfaz de Usuario (Frontend - Sección)
  El componente React responsable de la creación de secciones es [Sections.jsx](file:///d:/Workspace/utp/EduTrack/Frontend/src/components/Sections.jsx). Al enviar el formulario, se invoca `handleSectionSubmit`:

  ```javascript
  const handleSectionSubmit = async (e) => {
    e.preventDefault();
    if (!isAdmin) return;
    if (!courseId || !period || !code) {
      setError('Por favor complete todos los campos');
      return;
    }

    setError('');
    setSuccess('');

    try {
      const sectionData = { courseId: parseInt(courseId), period, code };
      if (isEditing) {
        await api.sections.update(selectedSectionId, sectionData);
        setSuccess('Sección actualizada con éxito');
      } else {
        await api.sections.create(sectionData);
        setSuccess('Sección creada con éxito');
      }
      setShowSectionModal(false);
      fetchData();
    } catch (err) {
      setError(err.message || 'Error al guardar la sección');
    }
  };
  ```

  El API helper en [api.js](file:///d:/Workspace/utp/EduTrack/Frontend/src/services/api.js) ejecuta:

  ```javascript
  export const api = {
    sections: {
      create: (data) => request('/sections', {
        method: 'POST',
        body: JSON.stringify(data),
      }),
      // ...
    }
  };
  ```

  ### 4. Capa de Negocio (Backend Service - Sección)
  La lógica de validación reside en [SectionService.java](file:///d:/Workspace/utp/EduTrack/Backend/src/main/java/com/utp/EduTrack/domain/service/SectionService.java). Verifica que no exista duplicidad de sección para el mismo curso, periodo y código:

  ```java
  public SectionDTO create(SectionCreateDTO request) {
      Course course = courseService.getCourseEntity(request.getCourseId());
      String period = request.getPeriod().trim();
      String code = request.getCode().trim().toUpperCase();

      if (sectionRepository.findByCourseIdAndPeriodAndCode(course.getId(), period, code).isPresent()) {
          throw new BusinessException("Ya existe una sección con ese curso, periodo y código");
      }

      Section section = Section.builder()
              .course(course)
              .period(period)
              .code(code)
              .build();

      return sectionMapper.toDto(sectionRepository.save(section));
  }
  ```

  ### 5. Capa de Persistencia (Repository & Entity - Sección)
  - **Repositorio**: [SectionRepository.java](file:///d:/Workspace/utp/EduTrack/Backend/src/main/java/com/utp/EduTrack/persistance/repository/SectionRepository.java), extiende de `JpaRepository<Section, Long>`. Utiliza el método personalizado:
    `Optional<Section> findByCourseIdAndPeriodAndCode(Long courseId, String period, String code);`
  - **Entidad**: [Section.java](file:///d:/Workspace/utp/EduTrack/Backend/src/main/java/com/utp/EduTrack/persistance/entity/Section.java), mapeada a la tabla `sections` con una restricción única compuesta (`uniqueConstraints` de `course_id`, `period` y `code`).
    ```java
    @Entity
    @Table(
            name = "sections",
            uniqueConstraints = @UniqueConstraint(columnNames = {"course_id", "period", "code"})
    )
    public class Section {
        @Id
        @GeneratedValue(strategy = GenerationType.IDENTITY)
        private Long id;

        @ManyToOne(fetch = FetchType.LAZY, optional = false)
        @JoinColumn(name = "course_id", nullable = false)
        private Course course;

        @ManyToOne(fetch = FetchType.LAZY)
        @JoinColumn(name = "teacher_id")
        private User teacher;

        @Column(nullable = false, length = 20)
        private String period;

        @Column(nullable = false, length = 10)
        private String code;
    }
    ```
    Se genera la inserción sql: `INSERT INTO sections (course_id, period, code) VALUES (?, ?, ?)`.

  ### 4. Capa de Negocio (Backend Service - Matrícula)
  La lógica y validaciones residen en [EnrollmentService.java](file:///d:/Workspace/utp/EduTrack/Backend/src/main/java/com/utp/EduTrack/domain/service/EnrollmentService.java). Valida que el usuario tenga rol `STUDENT`, que esté activo, y que no esté ya matriculado en la misma sección:

  ```java
  public EnrollmentDTO enroll(EnrollmentCreateDTO request) {
      User student = userService.getUserEntity(request.getStudentId());
      Section section = sectionService.getSectionEntity(request.getSectionId());

      if (student.getRole() != Role.STUDENT) {
          throw new BusinessException("Solo estudiantes pueden matricularse en una sección");
      }
      if (!student.isActive()) {
          throw new BusinessException("El estudiante debe estar activo");
      }
      if (enrollmentRepository.existsByStudentIdAndSectionId(student.getId(), section.getId())) {
          throw new BusinessException("El estudiante ya está matriculado en esta sección");
      }

      Enrollment enrollment = Enrollment.builder()
              .student(student)
              .section(section)
              .status(EnrollmentStatus.ENROLLED)
              .build();

      return enrollmentMapper.toDto(enrollmentRepository.save(enrollment));
  }
  ```

  ### 5. Capa de Persistencia (Repository & Entity - Matrícula)
  - **Repositorio**: [EnrollmentRepository.java](file:///d:/Workspace/utp/EduTrack/Backend/src/main/java/com/utp/EduTrack/persistance/repository/EnrollmentRepository.java), extiende `JpaRepository<Enrollment, Long>`. Usa:
    `boolean existsByStudentIdAndSectionId(Long studentId, Long sectionId);`
  - **Entidad**: [Enrollment.java](file:///d:/Workspace/utp/EduTrack/Backend/src/main/java/com/utp/EduTrack/persistance/entity/Enrollment.java), mapeada a la tabla `enrollments` con una clave única compuesta por `student_id` y `section_id`.
    ```java
    @Entity
    @Table(
            name = "enrollments",
            uniqueConstraints = @UniqueConstraint(columnNames = {"student_id", "section_id"})
    )
    public class Enrollment {
        @Id
        @GeneratedValue(strategy = GenerationType.IDENTITY)
        private Long id;

        @ManyToOne(fetch = FetchType.LAZY, optional = false)
        @JoinColumn(name = "student_id", nullable = false)
        private User student;

        @ManyToOne(fetch = FetchType.LAZY, optional = false)
        @JoinColumn(name = "section_id", nullable = false)
        private Section section;

        @Enumerated(EnumType.STRING)
        @Column(nullable = false, length = 20)
        @Builder.Default
        private EnrollmentStatus status = EnrollmentStatus.ENROLLED;
    }
    ```
    Se ejecuta la inserción sql: `INSERT INTO enrollments (student_id, section_id, status) VALUES (?, ?, ?)`.

  ### 6. Camino de Retorno (Común)
  1. El motor de persistencia guarda el registro y devuelve la entidad correspondiente (`Section` o `Enrollment`).
  2. El servicio mapea la entidad al DTO adecuado (`SectionDTO` o `EnrollmentDTO`) y lo devuelve al controlador.
  3. El controlador emite un `201 Created` con el JSON del DTO correspondiente.
  4. En frontend, `api.js` intercepta la respuesta satisfactoria, la parsea y resuelve la promesa.
  5. **Para Secciones**: `handleSectionSubmit` cierra el modal y ejecuta `fetchData()`, la cual llama a `api.sections.getAll()` y actualiza el estado `setSections(sectionsData)`, refrescando la tabla de secciones.
  6. **Para Matrículas**: `handleEnrollSubmit` cierra el modal e invoca a `fetchEnrollments(selectedSectionId)` pasando el ID de la sección actual. Dicha función ejecuta `api.enrollments.getBySectionId(...)` y actualiza el estado local de alumnos matriculados `setEnrollments(data || [])`, repintando inmediatamente el padrón de matriculados en la pantalla con el contador actualizado.
