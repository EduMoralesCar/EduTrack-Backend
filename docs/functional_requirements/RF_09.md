# RF 09: Registro de asistencia

- **Explicación del Requerimiento**:
  Este requerimiento permite a los docentes (`TEACHER`) y administradores (`ADMIN`) registrar y actualizar de forma semanal el estado de asistencia de cada estudiante matriculado en su sección. Las opciones de estado son: `PRESENTE`, `AUSENTE`, `TARDE` o `JUSTIFICADO`. Cada sesión semanal se mapea en el sistema a la fecha de inicio de semana (lunes correspondiente a la semana de ciclo 1 a 18). Al registrar una inasistencia (`AUSENTE`), se evalúa el riesgo académico del estudiante y se le despacha una notificación en tiempo real informándole de la falta.

  4. **Capa de Negocio (Backend Service)**:
     - **Nombre de Clase**: [AttendanceService.java](file:///d:/Workspace/utp/EduTrack/Backend/src/main/java/com/utp/EduTrack/domain/service/AttendanceService.java)
     - **Implementación del Método**: `recordAttendance`
     - **Mapeo de DTO**: Mapea el `AttendanceDTO` recibido a la entidad `Attendance`. Si la tupla de asistencia ya existe para esa fecha, sección y estudiante, la modifica y realiza limpieza preventiva de duplicados.
     - **Validaciones**:
       - Verifica la existencia de la sección y del estudiante matriculado.
       - Dispara de manera asíncrona la re-evaluación del riesgo académico del estudiante (`enrollmentService.evaluateAcademicRisk`).
       - Si el estado es `AUSENTE`, envía un mensaje informativo al estudiante mediante `notificationService.sendNotification`.
     - **Snippet de Código Java**:
       ```java
       @Transactional
       public AttendanceDTO recordAttendance(AttendanceDTO dto) {
           Section section = sectionRepository.findById(dto.getSectionId())
                   .orElseThrow(() -> new ResourceNotFoundException("Section not found"));
           User student = userRepository.findById(dto.getStudentId())
                   .orElseThrow(() -> new ResourceNotFoundException("Student not found"));

           List<Attendance> attendances = attendanceRepository.findBySectionIdAndStudentIdAndDate(
                   dto.getSectionId(), dto.getStudentId(), dto.getDate());
           Attendance attendance;

           if (!attendances.isEmpty()) {
               // Si existe, se actualiza el estado de la asistencia
               attendance = attendances.get(0);
               attendance.setStatus(dto.getStatus());
               // Limpieza de duplicados residuales si existieran
               if (attendances.size() > 1) {
                   for (int i = 1; i < attendances.size(); i++) {
                       attendanceRepository.delete(attendances.get(i));
                   }
               }
           } else {
               // Si es nueva, se construye usando el builder
               attendance = Attendance.builder()
                       .section(section)
                       .student(student)
                       .date(dto.getDate())
                       .status(dto.getStatus())
                       .build();
           }

           Attendance saved = attendanceRepository.save(attendance);
           dto.setId(saved.getId());

           try {
               enrollmentService.evaluateAcademicRisk(dto.getStudentId(), section.getId());
               if (dto.getStatus() == com.utp.EduTrack.persistance.entity.AttendanceStatus.AUSENTE) {
                   notificationService.sendNotification(student, "Se ha registrado una inasistencia en la sección " + section.getCode() + " para la fecha " + dto.getDate() + ".");
               }
           } catch (Exception e) {
               System.err.println("Warning evaluating risk or sending attendance notification: " + e.getMessage());
           }

           return dto;
       }
       ```

  5. **Capa de Persistencia (Repository & Entity)**:
     - **Repositorio**: [AttendanceRepository.java](file:///d:/Workspace/utp/EduTrack/Backend/src/main/java/com/utp/EduTrack/persistance/repository/AttendanceRepository.java)
       - **Métodos JPA Utilizados**: `findBySectionIdAndStudentIdAndDate(Long sectionId, Long studentId, LocalDate date)` para validar registros previos e impedir duplicidades y `save(Attendance attendance)` para persistir en BD.
     - **Entidad**: [Attendance.java](file:///d:/Workspace/utp/EduTrack/Backend/src/main/java/com/utp/EduTrack/persistance/entity/Attendance.java)
       - **Tabla de Base de Datos**: `attendances`
       - **Detalles de Registro/Actualización/Consulta**: Se ejecuta un query `SELECT` en la tabla `attendances` buscando coincidencias con `section_id`, `student_id` y `date`. Si no hay coincidencias se hace un `INSERT`, de lo contrario se ejecuta un `UPDATE` del campo `status` (`PRESENTE`, `AUSENTE`, `TARDE`, `JUSTIFICADO`) para la fila respectiva.

  6. **Camino de Retorno**:
     - El backend registra los cambios y retorna una respuesta JSON de la asistencia con HTTP `201 Created`.
     - La respuesta llega a la llamada asíncrona de `handleRecordAttendanceForWeek` en `Attendance.jsx`.
     - Se ejecuta `showNotification(...)` para desplegar un aviso flotante verde con la confirmación.
     - Automáticamente, se ejecuta una llamada a `api.attendance.getBySection(selectedSection)` para actualizar el arreglo del estado local `attendanceRecords` mediante `setAttendanceRecords(atts)`.
     - React detecta la actualización del estado y re-renderiza la grilla de semanas, actualizando visualmente el badge del estudiante y recalculando la tasa global de asistencia.
