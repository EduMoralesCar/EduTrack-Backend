# RF 01: Registro y Gestión de Usuarios (Plantilla Educativa)

Este documento sirve como guía educativa paso a paso para comprender cómo fluyen los datos en una arquitectura moderna basada en **React (Frontend)**, **Spring Boot (Backend)** y **PostgreSQL (Base de Datos)**.

---

## Explicación del Requerimiento (Genérico)
El objetivo de este requerimiento es permitir la administración (CRUD: Creación, Lectura, Actualización y Eliminación) de cuentas de usuario en el sistema. 
- Cada usuario consta de atributos básicos como identificador único, nombre de usuario, correo electrónico, contraseña (que debe almacenarse de forma segura) y rol/estado dentro del sistema.
- Se implementan controles de seguridad genéricos para asegurar que solo los usuarios autorizados (por ejemplo, con rol de Administrador) puedan realizar operaciones de escritura.

---

## Estructura del Flujo de Comunicación (Plantilla educativa)

### 1. Frontend (React)
El componente de la interfaz de usuario `[ComponenteReact]` maneja el estado del formulario de registro y realiza la petición al backend.

```javascript
// Ejemplo genérico en React ([ComponenteReact].jsx)
import React, { useState } from 'react';

export const [ComponenteReact] = () => {
  const [formData, setFormData] = useState({
    username: '',
    email: '',
    password: '',
    role: 'USER'
  });
  const [message, setMessage] = useState('');

  const handleChange = (e) => {
    setFormData({
      ...formData,
      [e.target.name]: e.target.value
    });
  };

  const handleSubmit = async (e) => {
    e.preventDefault();
    try {
      const response = await fetch('[EndpointUrl]', {
        method: 'POST',
        headers: {
          'Content-Type': 'application/json',
          'Authorization': 'Bearer [TokenJWT_Opcional]'
        },
        body: JSON.stringify(formData)
      });

      if (response.ok) {
        const data = await response.json();
        setMessage('Usuario registrado con éxito.');
      } else {
        setMessage('Error al registrar usuario.');
      }
    } catch (error) {
      setMessage('Error de conexión con el servidor.');
    }
  };

  return (
    <form onSubmit={handleSubmit}>
      <input name="username" onChange={handleChange} placeholder="Nombre de usuario" />
      <input name="email" onChange={handleChange} placeholder="Correo electrónico" />
      <input name="password" type="password" onChange={handleChange} placeholder="Contraseña" />
      <button type="submit">Registrar</button>
      {message && <p>{message}</p>}
    </form>
  );
};
```

### 2. Red (HTTP Call)
Cuando el usuario envía el formulario, se transmite un paquete HTTP a través de la red:

- **Método**: `POST`
- **URL**: `[EndpointUrl]` (ej. `http://localhost:8080/api/users`)
- **Cabeceras (Headers)**:
  - `Content-Type: application/json`
  - `Authorization: Bearer [TokenJWT]` (si el endpoint requiere autenticación previa)
- **Cuerpo (Payload JSON)**:
  ```json
  {
    "username": "usuario_ejemplo",
    "email": "ejemplo@correo.com",
    "password": "contrasenaSecretA123",
    "role": "USER"
  }
  ```

### 3. Controller (Spring Boot)
En el backend, `[ControllerClass]` recibe la petición HTTP, valida los datos de entrada y delega la lógica de negocio al servicio correspondiente.

```java
// Ejemplo genérico en Spring Boot ([ControllerClass].java)
package com.example.generic.controller;

import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/api/users")
@CrossOrigin(origins = "*")
public class [ControllerClass] {

    private final [ServiceClass] userService;

    public [ControllerClass]([ServiceClass] userService) {
        this.userService = userService;
    }

    @PostMapping
    public ResponseEntity<?> registerUser(@RequestBody [UserDTO] userDTO) {
        // En un entorno de producción se validarían los datos aquí (ej. @Valid)
        [UserDTO] createdUser = userService.createUser(userDTO);
        return new ResponseEntity<>(createdUser, HttpStatus.CREATED);
    }
}
```

### 4. Service
La capa de servicio `[ServiceClass]` contiene la lógica de negocio. Es donde se procesa la información antes de guardarla (por ejemplo, encriptando contraseñas o verificando reglas de negocio específicas).

```java
// Ejemplo genérico en Spring Boot ([ServiceClass].java)
package com.example.generic.service;

import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

@Service
public class [ServiceClass] {

    private final [RepositoryInterface] userRepository;
    private final [PasswordEncoderInterface] passwordEncoder;

    public [ServiceClass]([RepositoryInterface] userRepository, [PasswordEncoderInterface] passwordEncoder) {
        this.userRepository = userRepository;
        this.passwordEncoder = passwordEncoder;
    }

    @Transactional
    public [UserDTO] createUser([UserDTO] dto) {
        // Validación de negocio: Evitar duplicados
        if (userRepository.existsByEmail(dto.getEmail())) {
            throw new RuntimeException("El correo ya está registrado");
        }

        // Crear la entidad JPA a partir del DTO
        [JPAEntity] entity = new [JPAEntity]();
        entity.setUsername(dto.getUsername());
        entity.setEmail(dto.getEmail());
        
        // Encriptación de la contraseña antes de persistir
        String encryptedPassword = passwordEncoder.encode(dto.getPassword());
        entity.setPassword(encryptedPassword);
        entity.setRole(dto.getRole());
        entity.setActive(true);

        // Persistir en base de datos
        [JPAEntity] savedEntity = userRepository.save(entity);

        // Retornar el resultado convertido de nuevo a DTO
        return convertToDTO(savedEntity);
    }

    private [UserDTO] convertToDTO([JPAEntity] entity) {
        [UserDTO] dto = new [UserDTO]();
        dto.setId(entity.getId());
        dto.setUsername(entity.getUsername());
        dto.setEmail(entity.getEmail());
        dto.setRole(entity.getRole());
        return dto;
    }
}
```

### 5. Repository & Entity
La capa de datos mapea las clases de Java a tablas de base de datos relacional (PostgreSQL) usando JPA e interfaces de repositorio para realizar consultas SQL de forma transparente.

```java
// Entidad JPA ([JPAEntity].java)
package com.example.generic.model;

import jakarta.persistence.*;

@Entity
@Table(name = "[DatabaseTable_usuarios]")
public class [JPAEntity] {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, unique = true)
    private String username;

    @Column(nullable = false, unique = true)
    private String email;

    @Column(nullable = false)
    private String password;

    @Column(nullable = false)
    private String role;

    @Column(nullable = false)
    private boolean active;

    // Getters y Setters...
}
```

```java
// Repositorio JPA ([RepositoryInterface].java)
package com.example.generic.repository;

import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.stereotype.Repository;

@Repository
public interface [RepositoryInterface] extends JpaRepository<[JPAEntity], Long> {
    boolean existsByEmail(String email);
    boolean existsByUsername(String username);
}
```

### 6. Return Path
El flujo de retorno viaja en sentido inverso una vez completada la persistencia en la base de datos:

1. **PostgreSQL** confirma la inserción y asigna un ID generado automáticamente.
2. **`[RepositoryInterface]`** retorna el objeto `[JPAEntity]` guardado.
3. **`[ServiceClass]`** convierte el objeto guardado a un DTO de respuesta seguro (eliminando datos sensibles como la contraseña encriptada) y lo devuelve al controlador.
4. **`[ControllerClass]`** envuelve el DTO en un objeto `ResponseEntity`, configurando el estado HTTP a `201 Created` y serializándolo a formato JSON.
5. **Red**: El servidor web envía la respuesta HTTP JSON de vuelta al cliente.
6. **Frontend (React)**: La promesa del `fetch` se resuelve con éxito. El componente `[ComponenteReact]` recibe la respuesta JSON, actualiza el estado de la interfaz de usuario con un mensaje de confirmación y refresca las tablas o listas correspondientes si es necesario.
