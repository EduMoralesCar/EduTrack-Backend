# RF 01: Registro de usuarios

- **Explicación del Requerimiento**:
  Este requerimiento permite la creación y administración (registro, actualización, listado y eliminación) de cuentas de usuario en la plataforma. Solo los usuarios con rol `ADMIN` tienen privilegios para gestionar los usuarios del sistema. Cada usuario posee un nombre de usuario, correo electrónico, contraseña (encriptada), rol (`ADMIN`, `TEACHER` o `STUDENT`), y un estado de actividad (activo/inactivo).

- **Flujo de Código Paso a Paso**:

  ### 1. Interfaz de Usuario (Frontend)
  El componente React responsable de la interfaz es [Users.jsx](file:///d:/Workspace/utp/EduTrack/Frontend/src/components/Users.jsx). Cuando el administrador completa el formulario en el modal de creación y hace clic en el botón "Guardar", se ejecuta la función `handleSubmit`:

  ```javascript
  const handleSubmit = async (e) => {
    e.preventDefault();
    if (!username || !email || (!isEditing && !password)) {
      setError('Por favor complete todos los campos obligatorios');
      return;
    }

    setError('');
    setSuccess('');

    try {
      if (isEditing) {
        // Update user
        const updateData = { username, email, role, active: isActive };
        if (password) {
          updateData.password = password; // Only update password if filled
        }
        await api.users.update(selectedUserId, updateData);
        setSuccess('Usuario actualizado con éxito');
      } else {
        // Create user
        await api.users.create({ username, password, email, role });
        setSuccess('Usuario creado con éxito');
      }
      setShowModal(false);
      fetchUsers();
    } catch (err) {
      setError(err.message || 'Error al guardar el usuario');
    }
  };
  ```

  La llamada a la API utiliza el servicio encapsulado en [api.js](file:///d:/Workspace/utp/EduTrack/Frontend/src/services/api.js):

  ```javascript
  export const api = {
    users: {
      create: (data) => request('/users', {
        method: 'POST',
        body: JSON.stringify(data),
      }),
      // ...
    }
  };
  ```

  ### 2. Capa de Red
  - **Método HTTP**: `POST`
  - **URL Base**: `http://localhost:8081/api`
  - **Ruta (Path)**: `/users`
  - **Parámetros**: Ninguno.
  - **Cuerpo de la Petición (Request Body)**: Formato JSON.
    ```json
    {
      "username": "JuanPerez",
      "password": "PasswordSeguro123!",
      "email": "juan@edutrack.com",
      "role": "STUDENT"
    }
    ```

  ### 3. Controlador (Backend Controller)
  El controlador Spring Boot encargado de recibir la solicitud es [UserController.java](file:///d:/Workspace/utp/EduTrack/Backend/src/main/java/com/utp/EduTrack/web/controller/UserController.java). El endpoint está restringido a administradores utilizando Spring Security `@PreAuthorize("hasRole('ADMIN')")`:

  ```java
  @RestController
  @RequestMapping("/api/users")
  @RequiredArgsConstructor
  @SecurityRequirement(name = OpenApiConfig.BEARER_AUTH)
  @Tag(name = "Usuarios (ADMIN)", description = "RF01 - Solo rol ADMIN puede gestionar usuarios")
  public class UserController {

      private final UserService userService;

      @PostMapping
      @PreAuthorize("hasRole('ADMIN')")
      @Operation(summary = "Registrar usuario", description = "Campos: username, password, email, role (ADMIN|TEACHER|STUDENT)")
      public ResponseEntity<UserDTO> create(@Valid @RequestBody UserCreateDTO request) {
          return ResponseEntity.status(HttpStatus.CREATED).body(userService.create(request));
      }
  }
  ```

  ### 4. Capa de Negocio (Backend Service)
  La lógica de negocio se procesa en [UserService.java](file:///d:/Workspace/utp/EduTrack/Backend/src/main/java/com/utp/EduTrack/domain/service/UserService.java). El método `create` valida que las credenciales no estén repetidas, encripta la contraseña usando `PasswordEncoder` y mapea el objeto al DTO correspondiente:

  ```java
  public UserDTO create(UserCreateDTO request) {
      validateUniqueCredentials(request.getUsername(), request.getEmail(), null);

      User user = User.builder()
              .username(request.getUsername().trim())
              .email(request.getEmail().trim().toLowerCase())
              .password(passwordEncoder.encode(request.getPassword()))
              .role(request.getRole())
              .active(true)
              .build();

      return userMapper.toDto(userRepository.save(user));
  }

  private void validateUniqueCredentials(String username, String email, Long excludeId) {
      boolean usernameExists = excludeId == null
              ? userRepository.existsByUsername(username)
              : userRepository.existsByUsernameAndIdNot(username, excludeId);

      boolean emailExists = excludeId == null
              ? userRepository.existsByEmail(email)
              : userRepository.existsByEmailAndIdNot(email, excludeId);

      if (usernameExists) {
          throw new BusinessException("El nombre de usuario ya está registrado");
      }
      if (emailExists) {
          throw new BusinessException("El correo electrónico ya está registrado");
      }
  }
  ```

  ### 5. Capa de Persistencia (Repository & Entity)
  - **Repositorio**: [UserRepository.java](file:///d:/Workspace/utp/EduTrack/Backend/src/main/java/com/utp/EduTrack/persistance/repository/UserRepository.java), interfaz que extiende de `JpaRepository<User, Long>`. Utiliza los métodos `existsByUsername` y `existsByEmail` para las validaciones y el método heredado `save(User)` para registrar.
  - **Entidad**: [User.java](file:///d:/Workspace/utp/EduTrack/Backend/src/main/java/com/utp/EduTrack/persistance/entity/User.java), clase anotada con `@Entity` y mapeada a la tabla `users` en base de datos.
    ```java
    @Entity
    @Table(name = "users")
    @Getter
    @Setter
    @NoArgsConstructor
    @AllArgsConstructor
    @Builder
    public class User {
        @Id
        @GeneratedValue(strategy = GenerationType.IDENTITY)
        private Long id;

        @Column(nullable = false, unique = true, length = 50)
        private String username;

        @Column(nullable = false)
        private String password;

        @Column(nullable = false, unique = true, length = 120)
        private String email;

        @Enumerated(EnumType.STRING)
        @Column(nullable = false, length = 20)
        private Role role;

        @Column(name = "is_active", nullable = false)
        @Builder.Default
        private boolean active = true;
    }
    ```
    El motor de base de datos ejecuta una sentencia `INSERT INTO users (username, password, email, role, is_active) VALUES (?, ?, ?, ?, ?)` autogenerando la clave primaria `id`.

  ### 6. Camino de Retorno
  Una vez persistido en la base de datos:
  1. El repositorio devuelve la entidad guardada `User`.
  2. El servicio mapea la entidad a un `UserDTO` y lo retorna al controlador.
  3. El controlador responde con un código HTTP `201 Created` y el cuerpo JSON del `UserDTO`.
  4. La función `handleResponse` en `api.js` procesa la respuesta, verifica que sea exitosa y la deserializa.
  5. En `Users.jsx`, tras resolverse la promesa con éxito en `handleSubmit`, se llama a `setShowModal(false)` para cerrar el modal de registro y se ejecuta `fetchUsers()`.
  6. `fetchUsers()` realiza un llamado HTTP GET a `/users` mediante `api.users.getAll()`. Al obtener los datos, actualiza el estado local de React mediante `setUsers(data || [])`, lo que desencadena un re-renderizado de la interfaz mostrando el nuevo usuario en el listado.
