# Requerimiento Funcional: [RF_11 - Nombre del Requerimiento de Creación] (Plantilla Educativa)

## Explicación del Requerimiento (Genérico)
Esta plantilla describe el flujo de comunicación estándar para la **creación** de un nuevo recurso en el sistema. El objetivo es permitir que un usuario envíe datos a través de un formulario web y que estos se guarden de forma persistente en la base de datos PostgreSQL, garantizando validaciones previas y respondiendo con el recurso creado o un código de estado adecuado.

---

## Estructura del Flujo de Comunicación (Plantilla educativa)

### 1. Frontend (React)
El flujo comienza en el componente React `[ComponenteReactFormulario]`. Este componente maneja el estado local del formulario y maneja el evento de envío (`onSubmit`).

```javascript
// [ComponenteReactFormulario].jsx
import React, { useState } from 'react';

export const ComponenteFormulario = () => {
  const [formData, setFormData] = useState({
    campoUno: '',
    campoDos: ''
  });

  const handleChange = (e) => {
    setFormData({ ...formData, [e.target.name]: e.target.value });
  };

  const handleSubmit = async (e) => {
    e.preventDefault();
    try {
      // Invocación a la capa de servicio de red
      const response = await fetchCrearRecurso(formData);
      console.log('Recurso creado con éxito:', response);
    } catch (error) {
      console.error('Error al crear el recurso:', error);
    }
  };

  return (
    <form onSubmit={handleSubmit}>
      <input name="campoUno" value={formData.campoUno} onChange={handleChange} />
      <input name="campoDos" value={formData.campoDos} onChange={handleChange} />
      <button type="submit">Guardar</button>
    </form>
  );
};
```

### 2. Red (HTTP Call)
La llamada de red se realiza mediante un cliente HTTP (como `fetch` o `axios`) hacia el backend.

- **Método HTTP:** `POST`
- **Ruta/URL:** `[EndpointUrl]` (Ejemplo: `/api/v1/recursos`)
- **Cabeceras (Headers):** `Content-Type: application/json`
- **Cuerpo (Request Body):** JSON que representa los datos enviados desde el formulario.

```json
{
  "campoUno": "Valor de ejemplo 1",
  "campoDos": "Valor de ejemplo 2"
}
```

### 3. Controller (Spring Boot)
El controlador `[ControllerClass]` recibe la solicitud HTTP POST, valida los datos de entrada y delega la lógica al servicio.

```java
// [ControllerClass].java
@RestController
@RequestMapping("/api/v1/recursos")
public class ControllerClass {

    private final ServiceClass serviceClass;

    public ControllerClass(ServiceClass serviceClass) {
        this.serviceClass = serviceClass;
    }

    @PostMapping
    public ResponseEntity<JPAEntity> crearRecurso(@Valid @RequestBody DTORequest dto) {
        JPAEntity nuevoRecurso = serviceClass.guardarRecurso(dto);
        return new ResponseEntity<>(nuevoRecurso, HttpStatus.CREATED);
    }
}
```

### 4. Service
La capa de servicio `[ServiceClass]` contiene la lógica de negocio, mapea el DTO a la entidad y maneja las transacciones.

```java
// [ServiceClass].java
@Service
public class ServiceClass {

    private final RepositoryInterface repositoryInterface;

    public ServiceClass(RepositoryInterface repositoryInterface) {
        this.repositoryInterface = repositoryInterface;
    }

    @Transactional
    public JPAEntity guardarRecurso(DTORequest dto) {
        // Lógica de negocio (ej. validaciones adicionales, cálculos, asignaciones)
        JPAEntity entidad = new JPAEntity();
        entidad.setCampoUno(dto.getCampoUno());
        entidad.setCampoDos(dto.getCampoDos());
        
        return repositoryInterface.save(entidad);
    }
}
```

### 5. Repository & Entity
El repositorio `[RepositoryInterface]` interactúa con PostgreSQL usando JPA. La entidad `[JPAEntity]` mapea los campos a la estructura física de la base de datos.

```java
// [RepositoryInterface].java
@Repository
public interface RepositoryInterface extends JpaRepository<JPAEntity, Long> {
}

// [JPAEntity].java
@Entity
@Table(name = "tabla_recursos")
public class JPAEntity {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "campo_uno", nullable = false)
    private String campoUno;

    @Column(name = "campo_dos")
    private String campoDos;

    // Getters y Setters
}
```

### 6. Return Path
El flujo de retorno es el siguiente:
1. **Base de Datos:** Guarda el registro y retorna el ID autogenerado a `[RepositoryInterface]`.
2. **Repository:** Devuelve la instancia de la entidad guardada a `[ServiceClass]`.
3. **Service:** Completa la transacción y retorna el objeto a `[ControllerClass]`.
4. **Controller:** Envuelve el objeto en un `ResponseEntity` con estado HTTP `201 Created` y lo devuelve.
5. **Red:** Envía la respuesta JSON al cliente.
6. **Frontend (React):** `fetchCrearRecurso` resuelve la promesa y actualiza el estado de la UI (por ejemplo, mostrando un mensaje de éxito o redirigiendo al usuario).
