# Requerimiento Funcional: RF 02 - Autenticación de usuarios (Plantilla Educativa)

Este documento sirve como guía educativa paso a paso para comprender cómo fluyen los datos en un proceso de autenticación segura (Login) basado en **JSON Web Tokens (JWT)**.

---

## Explicación del Requerimiento (Genérico)
El objetivo de este requerimiento es validar la identidad de un usuario registrado a través de sus credenciales (nombre de usuario/correo y contraseña).
- Si las credenciales son válidas, el backend genera un token firmado digitalmente (JWT) que contiene información de identidad del usuario y sus roles asignados.
- El cliente (Frontend) almacena este token de manera segura y lo incluye en la cabecera de las solicitudes HTTP subsiguientes para acceder a recursos protegidos.

---

## Estructura del Flujo de Comunicación (Plantilla educativa)

### 1. Frontend (React)
El componente `[ComponenteReact]` maneja el formulario de inicio de sesión, recopila las credenciales y procesa la respuesta que contiene el token JWT.

```javascript
// Ejemplo genérico en React ([ComponenteReact].jsx)
import React, { useState } from 'react';

export const [ComponenteReact] = () => {
  const [credentials, setCredentials] = useState({ username: '', password: '' });
  const [error, setError] = useState('');

  const handleChange = (e) => {
    setCredentials({
      ...credentials,
      [e.target.name]: e.target.value
    });
  };

  const handleLogin = async (e) => {
    e.preventDefault();
    setError('');

    try {
      const response = await fetch('[EndpointUrl]', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify(credentials)
      });

      if (response.ok) {
        const data = await response.json();
        // Guardar el token en localStorage o de forma segura
        localStorage.setItem('token', data.token);
        window.location.href = '/dashboard';
      } else {
        setError('Credenciales inválidas');
      }
    } catch (err) {
      setError('Error al conectar con el servicio de autenticación');
    }
  };

  return (
    <form onSubmit={handleLogin}>
      <input name="username" onChange={handleChange} placeholder="Usuario" />
      <input name="password" type="password" onChange={handleChange} placeholder="Contraseña" />
      <button type="submit">Iniciar Sesión</button>
      {error && <p style={{ color: 'red' }}>{error}</p>}
    </form>
  );
};
```

### 2. Red (HTTP Call)
La petición HTTP viaja a través de la red transportando las credenciales crudas en el cuerpo de la petición.

- **Método**: `POST`
- **URL**: `[EndpointUrl]` (ej. `http://localhost:8080/api/auth/login`)
- **Cabeceras (Headers)**:
  - `Content-Type: application/json`
- **Cuerpo (Payload JSON)**:
  ```json
  {
    "username": "usuario_prueba",
    "password": "contrasenaCorrecta123"
  }
  ```

### 3. Controller (Spring Boot)
En el backend, `[ControllerClass]` expone el endpoint público de autenticación, mapea el payload de la petición y delega la verificación al servicio de seguridad.

```java
// Ejemplo genérico en Spring Boot ([ControllerClass].java)
package com.example.generic.controller;

import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/api/auth")
public class [ControllerClass] {

    private final [ServiceClass] authService;

    public [ControllerClass]([ServiceClass] authService) {
        this.authService = authService;
    }

    @PostMapping("/login")
    public ResponseEntity<?> authenticate(@RequestBody [LoginDTO] loginDTO) {
        [AuthResponseDTO] response = authService.login(loginDTO);
        return ResponseEntity.ok(response);
    }
}
```

### 4. Service
El servicio `[ServiceClass]` coordina la lógica de autenticación: verifica si el usuario existe, compara la contraseña encriptada y genera el token firmado si la comparación es exitosa.

```java
// Ejemplo genérico en Spring Boot ([ServiceClass].java)
package com.example.generic.service;

import org.springframework.stereotype.Service;

@Service
public class [ServiceClass] {

    private final [RepositoryInterface] userRepository;
    private final [PasswordEncoderInterface] passwordEncoder;
    private final [TokenProviderClass] tokenProvider;

    public [ServiceClass]([RepositoryInterface] userRepository, 
                          [PasswordEncoderInterface] passwordEncoder, 
                          [TokenProviderClass] tokenProvider) {
        this.userRepository = userRepository;
        this.passwordEncoder = passwordEncoder;
        this.tokenProvider = tokenProvider;
    }

    public [AuthResponseDTO] login([LoginDTO] loginDTO) {
        // 1. Buscar al usuario en la base de datos
        [JPAEntity] user = userRepository.findByUsername(loginDTO.getUsername())
                .orElseThrow(() -> new RuntimeException("Credenciales incorrectas"));

        // 2. Validar que la contraseña coincida con el hash almacenado
        if (!passwordEncoder.matches(loginDTO.getPassword(), user.getPassword())) {
            throw new RuntimeException("Credenciales incorrectas");
        }

        // 3. Validar si el usuario está activo
        if (!user.isActive()) {
            throw new RuntimeException("Usuario inactivo");
        }

        // 4. Generar el Token JWT
        String token = tokenProvider.generateToken(user.getUsername(), user.getRole());

        // 5. Retornar DTO con el token y datos básicos
        return new [AuthResponseDTO](token, user.getUsername(), user.getRole());
    }
}
```

### 6. Return Path
El flujo de retorno devuelve la respuesta al cliente:

1. **`[RepositoryInterface]`** retorna la entidad del usuario si la encuentra.
2. **`[ServiceClass]`** genera el token JWT codificando claims (como rol, fecha de expiración y emisor) firmado por una clave secreta.
3. El token se encapsula en una respuesta `[AuthResponseDTO]` y se pasa al controlador.
4. **`[ControllerClass]`** devuelve la respuesta HTTP con estado `200 OK` y el cuerpo JSON conteniendo el token.
5. **Red**: El payload JSON viaja al cliente.
6. **Frontend (React)**: El cliente recibe el token JWT, lo guarda en `localStorage` o en memoria segura, actualiza el contexto del usuario autenticado (rol, nombre) y redirige al usuario a la sección correspondiente del sistema. Las llamadas HTTP posteriores incluirán este token en el header `Authorization`.
